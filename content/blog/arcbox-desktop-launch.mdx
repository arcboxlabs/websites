Today we're releasing the public beta of ArcBox Desktop, a Docker Desktop alternative for macOS. The runtime is written in Rust, the desktop app is native SwiftUI, and the whole stack is open source under MIT/Apache-2.0.

![ArcBox Desktop running an nginx container with a terminal session](/blog/nginx-terminal.png)

## Why we built this

If Docker Desktop is the incumbent, OrbStack is the product we most directly compete with. It proved there was room for a much faster, lighter container experience on macOS.

OrbStack is also paid for business and commercial use. We wanted an open-source alternative that teams can use for free at work.

We also want to build VM-isolated sandboxes for running untrusted code. That means a real hypervisor layer, shared filesystems, networking, guest services, and VM lifecycle management. A runtime that can spin up Linux VMs on a Mac is already most of what a container tool needs, so the first product is a Docker Desktop alternative and the sandbox work builds on the same foundation.

## What we actually built

The runtime is about 116,000 lines of Rust across 26 crates. On macOS, the shipping path combines custom Rust systems code with Apple's Virtualization.framework, and the same codebase also includes the lower-level building blocks we're using for Linux/KVM and future work.

**Virtualization.framework bindings.** We wrote our own Rust bindings around Virtualization.framework: `objc2`-based wrappers plus low-level Objective-C messaging for VM configuration, shared directories, networking attachments, vsock, and VM lifecycle.

**Custom macOS networking datapath.** On macOS, ArcBox sets up a socketpair-backed `VZFileHandleNetworkDeviceAttachment`, then runs a Rust datapath task that handles DHCP, DNS forwarding, TCP via `smoltcp`, inbound port relay, and host socket proxying.

**Runtime foundations.** The broader runtime also includes custom `virtio`, filesystem, and networking crates: `virtio-net`, `virtio-blk`, `virtio-fs`, `virtio-console`, `virtio-vsock`, a VirtioFS/FUSE server used mainly for Linux/KVM paths, and batching/lock-free datapath primitives for packet processing. That's the foundation the macOS product grows out of, even when the Darwin launch path uses framework-backed devices.

**Guest agent.** A Rust guest agent runs inside the VM and talks to the host over vsock with protobuf messages. It mounts shared directories, reports runtime readiness, and exposes guest-side Docker and Kubernetes API proxies.

## Docker compatibility

ArcBox exposes a Docker-compatible socket on the host and proxies requests to a guest `dockerd` inside the VM.

```bash
arcbox docker enable   # creates and activates the "arcbox" Docker context
docker compose up      # works
docker build -t myapp .
docker run -p 8080:80 nginx
```

The source today covers the core workflows people expect from a Docker Desktop replacement: Docker CLI integration, Compose-compatible workflows, port forwarding, bind mounts, named volumes, and interactive exec/terminal flows.

Some less common API corners still need work, and x86/amd64 image emulation is not launch-ready yet. If you hit something broken, file an issue.

ArcBox also includes native `k3s` lifecycle support for local Kubernetes clusters.

## The desktop app

ArcBox Desktop is a native SwiftUI app. It talks to the ArcBox daemon over gRPC for ArcBox-specific operations and uses the Docker-compatible API for Docker resources.

At launch it covers Docker (containers, images, volumes, networks) and Kubernetes (pods, services). For Docker workloads, you get real-time log streaming with search and stdout/stderr filtering, an interactive terminal backed by `docker exec -it` and `docker run -it`, and local rootfs browsing when the app has a mount path it can resolve.

Machines and sandboxes share the same runtime foundation, but they are not part of this beta launch.

## Performance

There is already a boot-time benchmark harness in the repo, but we don't want to publish headline numbers until the broader benchmark suite and methodology are public and reproducible. For now, the right test is your own workload: builds, bind mounts, Compose projects, and everyday `docker` usage on your Mac.

## Open source

The runtime is at [github.com/arcboxlabs/arcbox](https://github.com/arcboxlabs/arcbox). The desktop app is at [github.com/arcboxlabs/arcbox-desktop](https://github.com/arcboxlabs/arcbox-desktop). Both are MIT/Apache-2.0, free for any use.

## What's coming

The same runtime also powers our sandbox work: VM-isolated environments for running untrusted workloads on the same foundation. If you've used E2B, that's the category we're aiming at on the sandbox side of ArcBox.

We're also continuing to expand the Linux/KVM side of the runtime and wire up Rosetta-backed x86 support that already has foundations in the codebase.
