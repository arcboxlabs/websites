Today we're releasing the public beta of ArcBox Desktop, a Docker Desktop alternative for macOS, built from scratch in Rust. It runs on Apple Silicon, it's open source (MIT/Apache-2.0), and local use is free.

![ArcBox Desktop running an nginx container with a terminal session](/blog/nginx-terminal.png)

## Why we built this

OrbStack proved that Docker Desktop on macOS can be much faster and lighter. But OrbStack is closed-source and requires a paid license for commercial use. We wanted something with the same ambition that's open source and free to use at work.

We also want to build VM-isolated sandboxes for running untrusted code. That requires a hypervisor layer, VirtIO devices, filesystem sharing, networking, essentially a full runtime. And a runtime that can spin up Linux VMs on a Mac is most of what a container tool needs. So the first product is a Docker Desktop alternative, and the sandbox work builds on the same foundation.

## What we actually built

The runtime is about 115,000 lines of Rust across 26 crates. Here's what's in there and why.

**Virtualization.framework bindings.** macOS has a hypervisor API, but there are no production-ready Rust bindings for it. We wrote our own: hand-rolled `objc_msgSend` calls using `objc2` for runtime primitives, covering VM configuration, device attachment, memory mapping, and the run loop.

**Custom VirtIO devices.** We implemented virtio-net, virtio-blk, virtio-fs, virtio-console, and virtio-vsock from the spec. Virtqueue batching and interrupt coalescing matter because each extra VM exit is measurable latency.

**Custom VirtioFS.** File sharing between macOS and the Linux VM is what makes bind mounts work. When you mount your project directory into a container, every file read and write crosses this layer. We wrote a FUSE passthrough filesystem with metadata caching and concurrent request dispatch (via rayon for parallel processing of the virtqueue). This is the single component we've spent the most time on.

**Custom NAT stack.** Userspace NAT engine with connection tracking, in-place packet translation, and a timer wheel for flow expiry. Our own DHCP and DNS servers. The network datapath uses a lock-free SPSC ring buffer to move packets between the VM and the host without contention.

**Guest agent.** A small Rust binary cross-compiled for Linux that runs inside the VM. It handles container lifecycle and PTY management, communicating with the host daemon over vsock with length-prefixed protobuf messages.

## Docker compatibility

ArcBox proxies the Docker Engine API to a dockerd instance running inside the VM.

```bash
arcbox docker enable   # points docker.sock to ArcBox
docker compose up      # works
docker build -t myapp .
docker run -p 8080:80 nginx
```

We estimate 95%+ of common use cases work: `docker build`, `docker compose`, port forwarding, bind mounts, named volumes, inter-container networking, `docker exec`. Some complex Compose projects run without changes.

What doesn't work yet: x86/amd64 image emulation (ARM64 only for now), and some less common API endpoints. If you hit something broken, file an issue.

ArcBox also ships with native k3s support for local Kubernetes clusters.

## The desktop app

ArcBox Desktop is a native SwiftUI app that talks to the Rust daemon over gRPC.

It covers Docker (containers, images, volumes, networks) and Kubernetes (pods, services). For containers, you get real-time log streaming with filtering, an interactive terminal, and a file browser for the container's rootfs. Linux machines and sandboxes are in the sidebar but not yet functional.

## Performance

We're not matching OrbStack on every metric yet, but we are ahead of Docker Desktop on engine startup and file-heavy workloads. We don't want to publish numbers without a reproducible benchmark suite to back them up. Try it with your own workloads and see.

## Open source

The runtime is at [github.com/arcboxlabs/arcbox](https://github.com/arcboxlabs/arcbox). The desktop app is at [github.com/arcboxlabs/arcbox-desktop](https://github.com/arcboxlabs/arcbox-desktop). Both are MIT/Apache-2.0.

We're a commercial company. Local use is free and stays free, no artificial limitations. The business comes from what's next: managed cloud sandboxes.

## What's coming

The sandbox runtime is under active development: VM-isolated microVMs for running untrusted code. Local sandboxes first, then a cloud API for running them on remote infrastructure. If you've used E2B, that's the direction, except you develop locally and deploy the same workloads to the cloud.

We're also working on Linux support (KVM backend) and Rosetta for x86 image emulation.

## Try it

The runtime is v0.3, the desktop app is v1.16. Still a beta, so expect rough edges.

```bash
brew tap arcboxlabs/tap
brew install arcbox
```
